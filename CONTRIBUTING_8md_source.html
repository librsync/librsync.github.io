3 Instructions and conventions for people wanting to work on librsync. Please

6 ## Pull requests
7 
8 Fixes or improvements in pull requests are welcome. Please:
9 
10 - [ ] Send small PRs that address one issues each.
11 
12 - [ ] Update `NEWS.md` to say what you changed.
13 
14 - [ ] Add a test as a self-contained C file in `tests/` that passes or fails,
15  and is hooked into `CMakeLists.txt`.
16 
17 - [ ] Keep the code style consistent with what's already there, especially in
18  keeping symbols with an `rs_` prefix.
19

21 ## NEWS
22 
23 [NEWS.md](NEWS.md) contains a list of user-visible changes in the library between
24 releases version. This includes changes to the way it's packaged,
25 bug fixes, portability notes, changes to the API, and so on.
26 
27 Add
28 and update items under a "Changes in X.Y.Z" heading at the top of
29 the file. Do this as you go along, so that we don't need to work
30 out what happened when it's time for a release.
31 
32 ## Tests
33 
34 Please try to update docs and tests in parallel with code changes.
35 
36 ## Releasing
37 
38 If you are making a new tarball release of librsync, follow this checklist:
39 
40 * NEWS.md - make sure the top "Changes in X.Y.Z" is correct, and the date is
41  correct.
42 
43 * `CMakeLists.txt` - version is correct.
44 
45 * `librsync.spec` - make sure version and URL are right.
46 
47 * Run `make all doc check` in a clean checkout of the release tag.
48 
49 Test results for builds of public github branches are at
50 https://travis-ci.org/librsync/librsync.
