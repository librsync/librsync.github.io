1 # IO callbacks {#api_callbacks}
2 
3 librsync jobs use IO callbacks to read and write files in pull mode.
4 It uses a "copy callback" to read data from the basis file for
5 "patch" operations in both push and pull mode.
6 
7 These callbacks
8 might write the data directly to a file or network connection, or they
9 can do some additional work such as compression or encryption.
10 
11 Callbacks are passed a `void *` baton, which is chosen by the application when
12 setting up the job. The baton can hold context or state for the
13 callback, such as a file handle or descriptor.
14 
15 There are three types of callbacks, for input, output, and a special
16 "copy callback" for random-access reads of the basis file when patching.
17 
18 ## Input and output callbacks
19 
20 Input and output callbacks are both of type ::rs_driven_cb. These
21 are used for all operations in pull mode only (see \ref api_pull).
22 
23 The callbacks are passed to rs_job_drive() and are called repeatedly
24 until the job completes, fails, or permanently blocks.
25 
26 Input and output callbacks can and must choose their own buffers, which they
27 provide as pointers to librsync as the job proceeds. There are many
28 possibilities:
29 
30  * The application may allocate a buffer when starting the job,
31  and shuffle data in and out of it as the job proceeds. As librsync
32  produces data in the output buffer, it is written out e.g. to a socket,
33  and the output pointer is then reset.
34 
35  * The application may allocate a single output buffer adequate to hold all
36  the output, and then the output callback need do nothing but let librsync
37  gradually consume it.
38 
39  * The input or output pointers might point into a mmap'd file.
40 
41  * The input and output buffers might be provided by some other library.
42 
43 The caller is responsible for freeing the buffer, and for remembering where
44 it previously asked librsync to write output.
45 
46 ## Input callbacks
47 
48 Input callbacks are passed a ::rs_buffers_s struct into which they can store
49 a pointer to the data they have read. Note that librsync does not allocate
50 the buffer, the caller must do so. The input callback should update
51 ::rs_buffers_s::next_in, ::rs_buffers_s::avail_in, and set ::rs_buffers_s::eof_in
52 if it's reached the end of the input.
53 
54 When an input callback reaches end-of-file and can return no more data, it
55 should return ::RS_INPUT_ENDED. If the callback has just a
56 little data left before end of file, then it should return that data
57 with ::RS_DONE. On the next call, unless the file has grown, it can
58 return ::RS_INPUT_ENDED.
59 
60 ## Output callbacks
61 
62 Output callbacks are also passed a ::rs_buffers_s struct. On the first call,
63 the output callback should store a pointer to its buffer into
64 ::rs_buffers_s::next_out, and the length into ::rs_buffers_s::avail_out. On
65 subsequent calls, librsync will have used some of this buffer and updated
66 those fields. The caller should then write out the used buffer space,
67 and possibly update the buffer to the place it wants new output to go.
68 
69 If the callback processes only part of the requested data, it should
70 still return ::RS_DONE.
71 In this case librsync will call the callback again later
72 until it either completes, fails, or blocks.
73 
74 The key thing to understand about ::rs_buffers_s is that the counts and
75 pointers are from librsync's point of view: the next byte, and the number
76 of bytes, that it should read or write.
77 
78 ## Copy callbacks
79 
80 Copy callbacks are used from both push-mode (rs_job_iter()) and pull-mode
81 (rs_job_drive()) invocations, only when doing a "patch" operation started by
82 rs_patch_begin().
83 
84 Copy callbacks have type ::rs_copy_cb.
85 
86 Copy callbacks are directly passed a buffer and length into which they
87 should write the data read from the basis file.
88 
89 ## Callback lifecycle
90 
91 IO callbacks are only called from within rs_job_drive() or
92 rs_job_iter().
93 
94 Different callbacks may be called several times in a
95 single invocation of rs_job_iter() or rs_job_drive().
96 
97 ## Return values
98 
99 Callbacks return a ::rs_result value to indicate success, an error, or
100 being blocked.
101 
102 If the callbacks return an error, that error will typically be passed
103 back to the application.
